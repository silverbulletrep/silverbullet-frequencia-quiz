# Story 5.1: Controle de Disparos e Contrato de Dados

**Epic:** [Epic 005: Disparo Automatico de Templates WhatsApp para Recuperacao de Vendas](./epic-005-whatsapp-recovery-dispatch.md)

---

## Status

**Draft**

---

## Story

**As a** operador do funil,
**I want** uma tabela de controle e um contrato de dados claro para os disparos de recuperacao WhatsApp,
**so that** o backend possa enviar templates pelo N8N sem duplicar mensagens e sem abordar leads que ja compraram.

---

## Acceptance Criteria

1. A migration cria a tabela `whatsapp_recovery_dispatches` sem alterar destrutivamente `funnel_events`, `funnel_leads` ou views existentes.
2. A tabela possui campos suficientes para auditoria: `id`, `lead_id`, `funnel_id`, `message_type`, `country`, `trigger_event_id`, `eligible_at`, `dispatched_at`, `n8n_status`, `n8n_response`, `created_at`, `updated_at`.
3. Existe constraint/indice unico que impede mais de um registro para o mesmo `(lead_id, message_type)`.
4. `message_type` aceita somente `multibanco_reminder`, `checkout_no_purchase` e `no_checkout`.
5. `n8n_status` aceita somente `pending`, `sent`, `failed` e `skipped_purchase_found`.
6. A migration inclui indices para as consultas esperadas do dispatcher: por `lead_id`, por `message_type`, por `n8n_status` e por `eligible_at`.
7. A story documenta o contrato minimo dos dados lidos em `funnel_events`, `funnel_leads` e `vw_funnel_lead_compact`.
8. A story documenta fallback de `country` quando `vw_funnel_lead_compact.country` estiver ausente.
9. A migration possui rollback manual documentado no proprio arquivo SQL ou em comentario adjacente.

## Dev Notes

### Previous Story Insights

Nao ha story anterior do Epic 005 implementada. O epic foi criado como Draft em `docs/stories/epic-005-whatsapp-recovery-dispatch.md` e define que Story 5.1 deve preparar banco/idempotencia antes do dispatcher backend. [Source: docs/stories/epic-005-whatsapp-recovery-dispatch.md#Stories]

### Data Models

`funnel_leads` usa chave primaria composta `(funnel_id, lead_id)` e armazena `attributes` e `metadata` como `jsonb`; estes campos podem conter contato e pais usados pelo dispatcher. [Source: ../Worker_Funil_01/db/schema.sql]

`funnel_events` possui `event_id`, `funnel_id`, `lead_id`, `event_type`, `event_timestamp`, `received_at`, `step_id`, `page_path`, `attributes`, `purchase` e `metadata`. A nova tabela deve referenciar `trigger_event_id` como `uuid`, mas nao deve criar foreign key obrigatoria se houver risco de eventos vindos de N8N antigo sem integridade completa. [Source: ../Worker_Funil_01/db/schema.sql]

`funnel_events.event_type` no schema local aceita `page_view`, `step_view`, `lead_qualified`, `desire_selected`, `step_progress`, `checkout_start`, `lead_identified` e `purchase`. O evento `PURCHASE_BILLET_PRINTED` aparece como dependencia do N8N Hotmart e deve ser validado no banco real antes de qualquer check constraint nova em tabelas existentes. [Source: ../Worker_Funil_01/db/schema.sql] [Source: hotmart-icN8N.md]

### Proposed Table Contract

Tabela: `public.whatsapp_recovery_dispatches`

Campos minimos:

| Field | Type | Notes |
|---|---|---|
| `id` | `uuid` | Primary key, default `gen_random_uuid()` |
| `lead_id` | `text` | Lead alvo do disparo |
| `funnel_id` | `text` | Funil de origem, esperado `quiz_frequencia_01` |
| `message_type` | `text` | `multibanco_reminder`, `checkout_no_purchase`, `no_checkout` |
| `country` | `text` | Pais enviado ao N8N; usar fallback documentado |
| `trigger_event_id` | `uuid null` | Evento que tornou o lead elegivel |
| `eligible_at` | `timestamptz` | Momento em que a janela de 10/25 min passou |
| `dispatched_at` | `timestamptz null` | Momento do envio real ao N8N |
| `n8n_status` | `text` | `pending`, `sent`, `failed`, `skipped_purchase_found` |
| `n8n_response` | `jsonb` | Resposta ou erro do N8N |
| `created_at` | `timestamptz` | Default `now()` |
| `updated_at` | `timestamptz` | Atualizado por trigger ou explicitamente |

### API / Backend Context

O backend Express registra rotas via `app.use('/api/...', routes)` em `BACKEND/api/app.ts`. A rota do dispatcher sera Story 5.2; esta story deve apenas criar a base de dados para ela. [Source: BACKEND/api/app.ts]

O padrao atual de N8N usa Axios com timeout configuravel em `BACKEND/api/lib/n8n.ts`. A tabela desta story deve armazenar o resultado que Story 5.2 receber desse sender. [Source: BACKEND/api/lib/n8n.ts]

### N8N / Purchase Context

O fluxo documentado de purchase envia dados ao N8N somente depois de validar Stripe/PayPal no backend. A recuperacao deve respeitar esses eventos de compra como bloqueio antes de disparar WhatsApp. [Source: SILVER-BULLET-AQUISICAO-FREQUENCIA/docs/architecture/n8n_flow_representation.md#1-stripe-flow] [Source: SILVER-BULLET-AQUISICAO-FREQUENCIA/docs/architecture/n8n_flow_representation.md#2-paypal-flow]

`purchaseN8N.md` registra eventos `purchase` em `funnel_events`; `hotmart-icN8N.md` registra eventos Hotmart com `attributes.payment_type`, incluindo o dado necessario para Multibanco/CASHPAYMENT. [Source: purchaseN8N.md] [Source: hotmart-icN8N.md]
