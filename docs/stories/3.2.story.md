## Story

**As a** usuário do quiz funnel,
**I want** que as páginas do quiz e páginas JS-heavy carreguem e renderizem rapidamente,
**so that** eu não veja uma tela vazia por vários segundos ao navegar entre quiz steps, processing, resultado e fim — eliminando o element render delay extremo (até 25s) que degrada a experiência e causa abandono.

---

## Acceptance Criteria

1. **Element render delay < 2s** em todos os quiz steps (QuizStep1–6) — medido via Lighthouse
2. **LCP score ≤ 4** nos quiz steps e na página `/resultado`
3. O **Suspense fallback** (AppFallback) é substituído por um **skeleton screen** que simula o layout da próxima página (quiz step layout: header + progress + question area)
4. Os **chunks dos quiz steps** são analisados e, se necessário, dependências pesadas são extraídas em chunks separados via code splitting granular
5. O `/vsl` mantém **element render delay ≤ 2s** (atualmente 1,650ms — monitorar que não regride)
6. O `/fim` tem **element render delay < 3s** (atualmente 8,560ms)
7. As **prefetch chains existentes** (QuizStep1→2→3→4→5→6→Processing) continuam funcionais e são otimizadas se necessário
8. **Nenhuma regressão** em checkout flow (Stripe, PayPal), VSL playback, UTM persistence e health probe
9. A avaliação de **InitialQuestions** (import eagerly em App.tsx L4) é documentada com decisão se deve ser lazy ou permanecer eager --

### Task 7 — Prerender Conservativo de /vsl a partir de /transition (AC: 5, 8)

> [!IMPORTANT]
> O objetivo é carregar o conteúdo pesado da VSL (scripts vturb, stream HLS) **enquanto o usuário ainda está em /transition**, para que ao navegar para /vsl o player já esteja pronto. **Nenhum evento de tracking (pixel, stepProgress, stepView) deve ser disparado** antes do usuário efetivamente entrar na página.

- [x] 7.1 Em `TransitionPage.jsx`, adicionar preload dos recursos críticos da VSL no `useEffect`:
  - `<link rel="preconnect" href="https://scripts.converteai.net">`
  - `<link rel="preconnect" href="https://cdn.converteai.net">`
  - `<link rel="dns-prefetch" href="https://images.converteai.net">`
  - `<link rel="dns-prefetch" href="https://api.vturb.com.br">`
  - `<link rel="preload" as="script" href="...smartplayer.js">`
  - `<link rel="preload" as="script" href="...player.js">`
  - Usar `requestIdleCallback` ou `setTimeout` para não bloquear o render da /transition
- [x] 7.2 Preload do stream HLS master playlist:
  - `<link rel="preload" as="fetch" href="...main.m3u8" crossorigin>`
  - Isso permite que o browser baixe o manifest do vídeo antes da navegação
- [x] 7.3 Prefetch do chunk JS de `VSL.jsx`:
  - Usar `import()` dinâmico para pré-baixar o código do componente (mesmo pattern do `usePrefetch` hook)
  - Trigger: `requestIdleCallback` após mount do TransitionPage
- [x] 7.4 **Garantir que tracking NÃO dispara no preload:**
  - Os eventos `stepProgress` e `stepView` do VSL (VSL.jsx L106-128) são disparados em `useEffect([], [])` — eles só executam quando o componente **monta no DOM**
  - Como estamos fazendo **preload de recursos** (scripts, fetch) e **NÃO montando o componente VSL**, os tracking events NÃO serão disparados
  - **Validar** em DevTools Network: ao estar em /transition, nenhuma request para a API de tracking deve aparecer
  - **Se futuramente migrar para Speculation Rules API prerender:** será necessário guardar tracking atrás de `document.prerendering` check e `prerenderingchange` event listener
- [x] 7.5 Prefetch do chunk JS via `<link rel="modulepreload">` como alternativa ao `import()`:
  - Avaliar se `modulepreload` é mais eficiente que `import()` (preload sem executar o módulo)
  - Decisão: usar o mais compatível com a arquitetura Vite existente
