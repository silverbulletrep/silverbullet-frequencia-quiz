## Story

**As a** operador tecnico do fluxo WhatsApp,
**I want** validar o contrato final backend -> N8N com template Meta resolvido,
**so that** o workflow envie a mensagem correta sem decidir regras de negocio no proprio N8N.

---

## Acceptance Criteria

1. O payload backend -> N8N inclui `template_id`, `meta_template_id`, `template_name`, `meta_language`, `template_variable_definitions` e `template_variable_values`.
2. O N8N continua apenas montando o payload Meta e nao decide `source_key`, `resolution_mode` ou `value_map`.
3. Existe validacao documentada para um caso de `pass_through` com valor presente.
4. Existe validacao documentada para um caso de `pass_through` usando `fallback_value`.
5. Existe validacao documentada para um caso de `mapped_value` com match.
6. Existe validacao documentada para um caso de `mapped_value` usando `fallback_value`.
7. Existe validacao documentada para `desire.response[0]` e `desire.response[1]` quando disponivel.
8. Se uma variavel obrigatoria nao puder ser resolvida, o backend registra status auditavel e o N8N nao recebe envio silencioso.
9. O runbook operacional fica atualizado para dry-run, primeiro envio real controlado e rollback.
10. O fluxo fim a fim continua sem roteamento estrutural por sexo.

### Previous Story Insights

Story 6.3 prepara a resolucao completa de template e variaveis no backend. Esta story fecha o contrato com o N8N e a validacao operacional fim a fim. [Source: docs/stories/6.3.recovery-backend-variable-resolution.md]

Story 5.3 ja criou um runbook operacional do dispatcher e uma estrategia de rollout controlado. O novo contrato deve aproveitar esse mesmo cuidado operacional. [Source: docs/stories/5.3.recovery-activation-validation.md]

### Existing N8N Context

O N8N atual ja consegue ler `template_variable_definitions` e `template_variable_values` para montar `components.body.parameters`, o que favorece manter a logica de negocio no backend. [Source: docs/stories/epic-006-recovery-template-routing-and-variable-resolution.md]

### Target Payload Shape

Payload minimo esperado:

```json
{
  "lead_id": "lead_x",
  "message_type": "checkout_no_purchase",
  "destination": "351...",
  "metadata": {
    "template_id": "local-template-uuid",
    "meta_template_id": "1614925649613959",
    "template_name": "recuperacao_portugal_base",
    "meta_language": "pt_PT",
    "template_variable_definitions": [
      { "token": "{{1}}", "index": 1, "label": "Nome", "required": true }
    ],
    "template_variable_values": {
      "{{1}}": "Maria"
    }
  }
}
```

### Validation Notes

- O N8N deve transformar o payload resolvido no formato Meta.
- O N8N nao deve conter `if/else` para escolher template por trigger.
- O backend deve continuar sendo a fonte da verdade para template, linguagem e valores finais.

- `POST /api/recovery/dispatch-due` com `dry_run` e `limit=1`
