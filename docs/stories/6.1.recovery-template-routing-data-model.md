**Ready for Review**

## Story

**As a** operador tecnico do recovery,
**I want** persistir o roteamento de template e os bindings de variaveis por rota,
**so that** o dashboard e o backend consigam resolver templates e placeholders sem logica hard-coded no N8N.

---

## Acceptance Criteria

1. Existe uma tabela `recovery_template_routes` para configurar qual template sera usado por `message_type + country`.
2. O schema nao possui coluna de sexo no roteamento.
3. Existe uma tabela `recovery_template_bindings` ligada a `recovery_template_routes` para persistir o binding de cada placeholder (`{{1}}`, `{{2}}`, etc.).
4. Cada binding persiste pelo menos `token`, `source_key`, `source_label`, `resolution_mode`, `fallback_value`, `required` e a configuracao de mapeamento dinamico.
5. `resolution_mode` aceita ao menos `pass_through` e `mapped_value`, protegido por constraint/check.
6. O schema suporta `value_map` por binding para transformar um valor bruto da `source_key` em um texto final do template.
7. O schema suporta `fallback_value` tanto para `mapped_value` sem match quanto para `pass_through` com valor `null`, vazio ou nao encontrado.
8. O schema documenta e limita as `source_key` de V1 aos campos de `vw_funnel_lead_compact`, incluindo subpaths JSONB de `desire`.
9. Existe unicidade operacional para evitar duas rotas ativas do mesmo `message_type + country`.
10. `message_templates` continua sendo o catalogo oficial de templates Meta, sem duplicacao de template payload nesta nova modelagem.

## Dev Notes

### Previous Story Insights

O Epic 006 ja bloqueou as decisoes principais desta capacidade: roteamento por `message_type + country`, fim do roteamento por sexo, `desire` como fonte oficial em `vw_funnel_lead_compact` e dois modos de resolucao (`pass_through` e `mapped_value`). [Source: docs/stories/epic-006-recovery-template-routing-and-variable-resolution.md]

### Existing Database Context

O dashboard ja possui `message_templates` como catalogo oficial de templates, com `meta_template_id`, `meta_language`, `meta_payload` e `variable_definitions`. A nova modelagem deve complementar esse catalogo, nao recria-lo. [Source: ../../Dashboard_2.0/dashbord/supabase/docs/SCHEMA.md]

`vw_funnel_lead_compact` ja expõe `name`, `email`, `phone`, `age`, `gender`, `country`, `auto_tag` e `desire jsonb`, com `desire.response` normalizado para ate dois valores. [Source: ../../Dashboard_2.0/dashbord/supabase/docs/SCHEMA.md]

### Recommended Data Shape

Persistir os bindings por `route_id` evita acoplamento excessivo ao template puro e permite reutilizar o mesmo `message_templates.template_id` em mais de uma rota com configuracoes de variavel diferentes.

Campos sugeridos para `recovery_template_routes`:

- `route_id`
- `message_type`
- `country`
- `template_id`
- `is_active`
- `created_at`
- `updated_at`

Campos sugeridos para `recovery_template_bindings`:

- `binding_id`
- `route_id`
- `token`
- `source_key`
- `source_label`
- `resolution_mode`
- `value_map`
- `fallback_value`
- `required`
- `created_at`
- `updated_at`

### Source Key Constraints

As `source_key` de V1 devem sair exclusivamente de `vw_funnel_lead_compact`. O schema pode usar `text` para persistencia, mas a aplicacao deve validar o conjunto permitido.

### Security / Operational Notes

- Nao reintroduzir sexo como coluna de roteamento.
- Nao duplicar `meta_payload` nesta nova modelagem.
- Manter a mudanca rollbackable por drop/desativacao de rotas novas, sem mexer no catalogo legado de templates.

### Testing

Validacoes esperadas nesta story:

- Revisao manual da migration.
- Teste de unicidade para rota ativa por `message_type + country`.
- Teste de unicidade de `token` por rota.
- Teste de `check constraint` para `resolution_mode`.
- Teste de persistencia de `value_map` e `fallback_value`.
- Atualizacao de `SCHEMA.md` no dashboard.
