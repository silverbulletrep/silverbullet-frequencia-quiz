## Story

**As a** operador do dashboard de recovery,
**I want** configurar a rota de template e os bindings de variaveis por placeholder,
**so that** cada trigger/pais use o template certo com resolucao de valores sem ajuste manual no N8N.

---

## Acceptance Criteria

1. Existe uma nova rota dedicada `/recovery-template-routing` no dashboard.
2. A tela permite configurar qual `template_id` vai para cada combinacao `message_type + country`.
3. A tela exibe um catalogo fixo de `source_key` com texto amigavel para o operador.
4. Para cada placeholder do template (`{{1}}`, `{{2}}`, etc.), o operador consegue configurar `source_key`, `resolution_mode`, `required` e `fallback_value`.
5. Quando `resolution_mode = mapped_value`, o operador consegue cadastrar pares `valor bruto -> texto final`.
6. Quando `resolution_mode = pass_through`, o operador consegue cadastrar `fallback_value` para ausencia de valor da `source_key`.
7. A UI deixa claro que `name + pass_through` sera tratado no backend como primeiro nome com caixa normalizada.
8. A UI nao oferece configuracao de roteamento por sexo.
9. Formularios invalidos nao podem ser salvos.
10. A configuracao serializada pela UI segue exatamente o contrato esperado pelo backend.

## Dev Notes

### Previous Story Insights

Esta story depende da modelagem da Story 6.1 para persistir rotas e bindings. O epic ja travou a existencia da rota dedicada `/recovery-template-routing` e a remocao do roteamento por sexo. [Source: docs/stories/6.1.recovery-template-routing-data-model.md] [Source: docs/stories/epic-006-recovery-template-routing-and-variable-resolution.md]

### Existing Dashboard Context

O dashboard ja tem uma pagina `/templates` e uma rota `/ai-recovery`, mas a segunda nao deve ser reutilizada para esta capacidade. A navegacao atual esta em `src/app/router.tsx`. [Source: ../../Dashboard_2.0/dashbord/src/app/router.tsx]

`templateVariableService.ts` hoje modela apenas `token`, `index`, `label`, `example` e `required`. Esta story deve evoluir ou complementar esse contrato para incluir `source_key`, `resolution_mode`, `fallback_value` e `value_map`. [Source: ../../Dashboard_2.0/dashbord/src/services/templateVariableService.ts]

### UX Constraints

- As opcoes de `source_key` devem ser fechadas e amigaveis.
- O operador deve ver claramente a diferenca entre:
  - `pass_through`
  - `mapped_value`
- `fallback_value` deve ser configuravel nos dois modos, mas com semanticas diferentes.
- Sexo nao deve aparecer como criterio de roteamento.

### Suggested Source Key Labels

- `name` -> `Nome completo`
- `email` -> `E-mail`
- `phone` -> `Telefone`
- `age` -> `Idade`
- `gender` -> `Genero`
- `country` -> `Pais`
- `auto_tag` -> `Auto tag atual`
- `desire.question` -> `Pergunta do desejo`
- `desire.response[0]` -> `Primeiro desejo`
- `desire.response[1]` -> `Segundo desejo`

### Testing

Validacoes esperadas nesta story:

- Teste de renderizacao da nova rota.
- Teste de formulario com rota valida.
- Teste de formulario invalido sem `source_key`.
- Teste de `mapped_value` com `value_map`.
- Teste de `pass_through` com `fallback_value`.
- Teste de serializacao igual ao contrato esperado pelo backend.
- Teste visual confirmando ausencia de opcoes de roteamento por sexo.
