# Story 6.2: Dashboard Configuration for Recovery Template Routing and Variable Bindings

## Status

**In Progress**

## Story

**As a** operador do dashboard de recovery,
**I want** configurar a rota de template e os bindings de variaveis por placeholder,
**so that** cada trigger/pais use o template certo com resolucao de valores sem ajuste manual no N8N.

---

## Acceptance Criteria

1. Existe uma nova rota dedicada `/recovery-template-routing` no dashboard.
2. A tela permite configurar qual `template_id` vai para cada combinacao `message_type + country`.
3. A tela exibe um catalogo fixo de `source_key` com texto amigavel para o operador.
4. Para cada placeholder do template (`{{1}}`, `{{2}}`, etc.), o operador consegue configurar `source_key`, `resolution_mode`, `required` e `fallback_value`.
5. Quando `resolution_mode = mapped_value`, o operador consegue cadastrar pares `valor bruto -> texto final`.
6. Quando `resolution_mode = pass_through`, o operador consegue cadastrar `fallback_value` para ausencia de valor da `source_key`.
7. Quando `resolution_mode = mapped_value`, o operador consegue cadastrar `fallback_value` para quando nenhum match for encontrado.
8. A UI deixa claro que `name + pass_through` sera tratado no backend como primeiro nome com caixa normalizada.
9. A UI nao oferece configuracao de roteamento por sexo.
10. Formularios invalidos nao podem ser salvos.
11. A configuracao serializada pela UI segue exatamente o contrato esperado pelo backend.
12. A nova tela reutiliza o layout, primitives e padroes visuais ja existentes do dashboard, sem criar um visual paralelo fora do design system atual.
13. A implementacao reutiliza componentes existentes sempre que possivel antes de criar novos blocos especificos de `RecoveryTemplateRouting`.

### Design System Guardrails

- Implementar sobre MUI e tokens/cores ja herdados do tema ativo do dashboard; nao introduzir novo sistema paralelo de cores, tipografia ou spacing.
- Priorizar composicao com primitives existentes do dashboard em vez de CSS modules novos para layout basico de formularios.
- Componentes novos devem nascer no namespace `src/components/RecoveryTemplateRouting/` apenas quando nao houver equivalente claro em `Templates/`.
- A pagina deve parecer uma extensao natural de `/templates`, nao uma ferramenta separada.
- Estados obrigatorios da interface:
  - loading inicial
  - empty state sem rotas
  - erro de carregamento/salvamento
  - sucesso apos salvar
  - aviso de configuracao incompleta
- O fluxo deve reduzir carga cognitiva com agrupamento claro:
  - secao 1: rota (`message_type + country + template_id`)
  - secao 2: placeholders detectados
  - secao 3: configuracao de binding
  - secao 4: revisao/validacao antes de salvar
- Evitar expor JSON cru ao operador como interface principal; `value_map` deve ser editado em UI orientada a linhas/campos.
- Responsividade deve seguir os breakpoints ja usados no dashboard e preservar usabilidade em tablet/mobile.

### UX Constraints

- As opcoes de `source_key` devem ser fechadas e amigaveis.
- O operador deve ver claramente a diferenca entre:
  - `pass_through`
  - `mapped_value`
- `fallback_value` deve ser configuravel nos dois modos, mas com semanticas diferentes.
- Sexo nao deve aparecer como criterio de roteamento.

### Suggested Source Key Labels

- `name` -> `Nome completo`
- `email` -> `E-mail`
- `phone` -> `Telefone`
- `age` -> `Idade`
- `gender` -> `Genero`
- `country` -> `Pais`
- `auto_tag` -> `Auto tag atual`
- `desire.question` -> `Pergunta do desejo`
- `desire.response[0]` -> `Primeiro desejo`
- `desire.response[1]` -> `Segundo desejo`
