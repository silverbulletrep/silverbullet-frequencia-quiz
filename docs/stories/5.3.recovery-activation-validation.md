## Story

**As a** operador tecnico do funil,
**I want** ativar o dispatcher de recuperacao com um intervalo controlado, dry-run e rollout seguro,
**so that** possamos enviar templates WhatsApp automaticamente sem disparar lote acidental ou impactar compras existentes.

---

## Acceptance Criteria

1. O backend possui um mecanismo interno de ativacao por intervalo, desligado por padrao via `RECOVERY_DISPATCH_ENABLED`.
2. O intervalo usa env `RECOVERY_DISPATCH_INTERVAL_MS` e respeita `RECOVERY_DISPATCH_LIMIT`.
3. O mecanismo interno reutiliza a mesma logica de `POST /api/recovery/dispatch-due` ou o mesmo service `recoveryDispatcher`, sem duplicar regra de negocio.
4. A ativacao nao roda quando `RECOVERY_DISPATCH_ENABLED` nao for exatamente `true`.
5. Existem env vars documentadas: `N8N_META_TEMPLATE_WEBHOOK_URL`, `RECOVERY_DISPATCH_SECRET`, `RECOVERY_DISPATCH_ENABLED`, `RECOVERY_DISPATCH_INTERVAL_MS`, `RECOVERY_DISPATCH_LIMIT`.
6. Existe runbook de validacao em `dry_run` para 1 candidato de cada regra: Multibanco, IC sem compra, Sem IC.
7. Existe procedimento de primeiro disparo real com `limit=1`.
8. O payload esperado pelo workflow N8N/Meta esta documentado para o time configurar templates por `message_type + country/language`.
9. Existe rollback simples documentado: desligar `RECOVERY_DISPATCH_ENABLED` e/ou remover `N8N_META_TEMPLATE_WEBHOOK_URL`.
10. Logs permitem rastrear `lead_id`, `message_type`, `eligible_at`, `n8n_status` e resposta N8N sem expor secrets.
11. Build TypeScript do backend passa com `npm run build:api`.

### Cron Definition for This Project

Neste projeto, "cron" significa um agendador interno do backend Node, provavelmente `setInterval`, protegido por env vars. Nao e um cron de sistema operacional nesta story. O objetivo e acordar o dispatcher periodicamente para verificar leads que completaram 10 ou 25 minutos. [Source: docs/stories/epic-005-whatsapp-recovery-dispatch.md#Stories]

### Backend Startup Context

`BACKEND/api/server.ts` sobe o Express com `app.listen(PORT)` e registra handlers de `SIGTERM`/`SIGINT`. O scheduler deve ser iniciado no bootstrap do servidor, nao em `app.ts`, para evitar side effects quando o app for importado em testes. [Source: BACKEND/api/server.ts] [Source: BACKEND/api/app.ts]

### Operational Env Vars

Env vars obrigatorias/recomendadas:

| Env Var | Purpose | Default recommended |
|---|---|---|
| `N8N_META_TEMPLATE_WEBHOOK_URL` | Webhook N8N que dispara template Meta/WhatsApp | none; se ausente, nao enviar |
| `RECOVERY_DISPATCH_SECRET` | Segredo para endpoint manual `dispatch-due` | none; obrigatorio para endpoint |
| `RECOVERY_DISPATCH_ENABLED` | Liga/desliga scheduler interno | `false` |
| `RECOVERY_DISPATCH_INTERVAL_MS` | Intervalo entre execucoes | `60000` |
| `RECOVERY_DISPATCH_LIMIT` | Max candidatos por ciclo | `20` |
| `N8N_TIMEOUT_MS` | Timeout HTTP existente para N8N | usar padrao existente se reaproveitado |

### N8N Workflow Contract

O N8N deve escolher template Meta usando `message_type` e `country`/`language`. O backend deve enviar pelo menos:

```json
{
  "lead_id": "lead_x",
  "message_type": "multibanco_reminder",
  "country": "PT",
  "language": "pt",
  "phone": "351...",
  "email": "lead@email.com",
  "name": "Nome",
  "funnel_id": "quiz_frequencia_01",
  "trigger": {
    "event_id": "uuid",
    "event_type": "PURCHASE_BILLET_PRINTED",
    "event_timestamp": "2026-05-01T12:00:00.000Z",
    "payment_type": "CASHPAYMENT"
  }
}
```

n8n Webhook node recebe payload externo como gatilho; Respond to Webhook pode devolver `accepted`, `sent` ou erro para o backend registrar. [Source: docs/stories/epic-005-whatsapp-recovery-dispatch.md#Codebase--Documentation-Audit-Summary]

### Validation Sequence

A validacao deve seguir esta ordem:

1. `dry_run` com `lead_id` especifico ou `limit=1`.
2. Conferir candidato e telefone.
3. Conferir ausencia de `purchase`.
4. Conferir `message_type`, `country` e `language`.
5. Envio real com `limit=1`.
6. Conferir N8N.
7. Conferir `whatsapp_recovery_dispatches`.
8. So entao aumentar `RECOVERY_DISPATCH_LIMIT`.
