# Story 3.4: Fix Broken Backgrounds on Deploy (Processing & Resultado)

## 🎯 Objetivo
Corrigir o layout quebrado (frestas/bordas brancas) no background das páginas `/processing` e `/resultado` em ambiente de deploy (pós-build).

## 🔍 Insights da Investigação e Causa Raiz

Durante a auditoria, identificamos três motivos principais que, em conjunto, causam o problema visual relatado:

1. **Conflito de Viewport com o `<PageTransition>` e `<AuthorityHeader>`**
   Com a introdução do Epic 003, as rotas foram englobadas no `<PageTransition>`, que é configurado como um container flexível com `min-height: 100dvh`. Como o `<AuthorityHeader>` é renderizado _acima_ do PageTransition, ele "empurra" o conteúdo para baixo. O tamanho total da tela rolável então se torna algo como `100dvh + 60px`.

2. **Uso Rígido de Unidades `vw` e `vh`**
   Múltiplas páginas (como a classe `.background3` em `Resultado.module.scss`) possuem estilização com medidas absolutas de viewport: `width: 100vw; height: 100vh;` (ou `min-height: 100vh;`). 
   Como esses containers não são instruídos a preencher o elemento pai (`flex: 1`), o background é desenhado apenas até atingir `100vh` a partir do topo deles. Quando se faz a rolagem (scroll) ou quando o navegador calcula a barra (compensando o header), a faixa excedente na parte inferior fica sem o background da página, revelando a cor branca original do `#root`/`body`.

3. **Inconsistência entre Local e Deploy (O motivo de `/processing` funcionar localmente)**
   A página `/processing` está visualmente correta no ambiente local (*Print 02*) devido ao fato de que, no commit `97fde4a`, os estilos de `ProcessingPage.module.scss` foram consertados para usar `width: 100%; flex: 1;`. No entanto, **o build atualmente servido em `/deploy` não inclui as alterações deste commit (está desatualizado frente à branch local)**.
   No deploy, a versão antiga do CSS (onde consta `width: 100vw; height: 100vh;`) ainda está rodando, quebrando a tela conforme visto no *Print 01*.

## ✅ Acceptance Criteria (Critérios de Aceite)
- [ ] As páginas dependentes de background dinâmico (`/processing`, `/resultado`) devem ocupar 100% da área contida pelo `<PageTransition>`, sem gerar white-spaces durante o scroll.
- [ ] O deploy e assets criados no Hostinger (`/deploy`) devem estar sincronizados com as atualizações que removem unidades rígidas de `100vw/100vh` substituindo-as por mecanismos flex.
- [ ] O `<PageTransition>` deve manter comportamentos coerentes e não forçar cortes abruptos independentemente do browser.

## 📋 Tasks

- [ ] Auditar outras páginas do Funil (`QuizStep*`, `AgeSelection*`, `Success*`) para certificar que o uso generalizado de `100vh`/`100vw` não as quebre silenciosamente pelo mesmo motivo.
